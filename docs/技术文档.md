# DA数据清洗业务AI应用 技术文档

> 给开发者和技术同事看的内部文档，说清楚这个工具是怎么工作的、用了什么技术、以及如果要改东西从哪里下手。

---

## 一、整体架构概览

```
浏览器 (HTML + JS)
    ↓ HTTP (JSON)
Flask Web 服务器 (Python)
    ↓
DuckDB (嵌入式数据库)     DeepSeek API (AI 代码生成)
    ↓
本地文件系统 (SQLite 文件、JSON、JSONL)
```

这是一个**纯本地**工具，没有外部数据库，没有微服务，没有容器化。所有数据都存在用户电脑的 `temp/` 目录下。

**为什么这么设计？** 审计数据敏感，不能上传到第三方服务器。DuckDB 是嵌入式的，随 Flask 进程启动，数据文件就在本地硬盘上，审计结束删除即可。

---

## 二、技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| Web 框架 | Flask 3.x | 轻量、灵活、Python 生态成熟 |
| 数据分析引擎 | DuckDB 1.5+ | 嵌入式列式数据库，处理百万行 CSV 不卡顿，SQL 标准兼容好 |
| AI 代码生成 | DeepSeek API | 通过自然语言生成 SQL，替代手写查询 |
| 前端 | 原生 HTML + CSS + JS | 零构建步骤、无框架依赖、修改直接生效 |
| 代码编辑器 | CodeMirror 5 | SQL 语法高亮、主题支持 |
| 会话管理 | Flask-Session (文件系统) | 服务端会话，支持多个会话互不干扰 |
| 文件上传 | Flask 原生 | 简单可靠，配合 Werkzeug 安全文件名 |

**为什么不选 XX？**
- **不用 React/Vue**：工具功能聚焦，页面数少，原生 JS 够用，省掉构建步骤
- **不用 PostgreSQL**：需要安装服务端进程，对审计员不友好；DuckDB 零运维
- **不用 Pandas**：Pandas 100 万行以上内存扛不住，DuckDB 列式引擎快 10-100 倍

## 三、数据流（关键路径）

### 3.1 上传 → 查询 完整链路

```
用户上传 CSV/XLSX
    │
    ▼
app.py: upload_file()
    ├─ 文件保存到 temp/ 目录
    ├─ 读取前100行做字段检测和预览
    ├─ 清空之前的 DuckDB 会话（避免新旧数据混淆）
    └─ 跳转到字段映射页
    │
    ▼
app.py: configure_fields()
    ├─ 接收前端发来的字段映射关系
    ├─ 启动 DuckDB 导入（CSV 直接导入 / XLSX 走 pandas 桥接）
    │   ├─ CSV: duckdb_engine.import_csv()
    │   │   ├─ CREATE TABLE AS SELECT * FROM read_csv(...)
    │   │   ├─ 应用重命名映射（若有）
    │   │   └─ 保留未映射字段（不丢失原始数据）
    │   └─ XLSX: duckdb_engine.import_xlsx()
    │       ├─ pandas.read_excel() → 注册到 DuckDB
    │       └─ 应用重命名映射
    ├─ 自动保存映射历史（按文件名+大小哈希索引）
    └─ 跳转到查询页
    │
    ▼
app.py: generate_code()
    ├─ 读取 DuckDB 表结构（字段名+类型）作为上下文
    ├─ 调用 DeepSeek API 生成 SQL
    ├─ 记录审计轨迹（JSONL 文件）
    ├─ 保存查询历史到 session
    └─ 返回 SQL 给前端
    │
    ▼
app.py: execute_code()
    ├─ 在 DuckDB 上执行 SQL（只读模式）
    ├─ 分页返回结果（默认每页 20 行）
    ├─ 记录审计轨迹
    └─ 前端渲染成表格
```

### 3.2 DuckDB 会话生命周期

```
每个上传会话都有一个独立的 .db 文件
路径: temp/db/session_<uuid>.db

创建时机: 字段映射配置成功时 (configure_fields)
销毁时机:
  ├─ 用户重新上传 → 清空 session → cleanup_session_db()
  ├─ 访问上传页 → cleanup_session_db()
  ├─ 上传失败 → 删除 .db 文件
  └─ 应用启动 → cleanup_stale_files() 删除超过24h的孤儿文件

生命周期内:
  └─ 查询、完整性测试、取样、预设规则都操作同一个表 "data"
      若有科目余额表，额外操作 "balance_data" 表
```

## 四、核心模块说明

### 4.1 `modules/duckdb_engine.py` — DuckDB 引擎

这是数据层的核心。封装了 DuckDB 的连接管理、导入、查询。

```
class DuckDBEngine:
    ├─ init_session(db_path)     → 创建/打开 .db 文件
    ├─ import_csv(path, table,   → CSV 导入（核心路径）
    │   rename_mapping, cols)
    │   ├─ 3 种分支:
    │   │   A. file_column_mapping  → 只导入指定列（建临时表→SELECT→DROP→RENAME）
    │   │   B. rename_mapping       → 导入全部列，重命名匹配的列
    │   │   C. 无映射               → 全部导入，原始列名
    │   └─ 读取 CSV 时用 all_varchar=true 避免类型推断错误
    │       再用 CAST 转目标类型
    ├─ import_xlsx(path, table,  → XLSX 通过 pandas 桥接
    │   rename_mapping, cols)
    ├─ execute(sql)              → 执行查询，返回 dict 列表
    ├─ execute_paginated(sql,    → 分页查询
    │   page, page_size)
    ├─ get_schema(table)         → 获取字段名+类型
    ├─ get_total_rows(table)     → COUNT(*)
    ├─ close()                   → 关闭连接
    └─ cleanup()                 → 关闭+删除 .db 文件
```

**导入 CSV 时的关键细节：**
- `read_csv` 用 `all_varchar=true` 和 `quote='"'`，防止分隔符/引号问题
- Branch B（rename_mapping）是默认路径，保留所有非映射字段
- Branch A（file_column_mapping）是历史遗留，它会丢弃未映射列，已不再使用
- 导入错误会捕获 DuckDB 的行号信息，拼接成"第 X 行数据有问题"给用户

### 4.2 `modules/ai_codegen.py` — AI 代码生成

调用 DeepSeek API 生成 SQL，不走 Python 代码路径。

```
class AICodeGenerator:
    ├─ generate(query, schema, preview)  → 核心方法
    │   ├─ 构建 system prompt（含 DuckDB SQL 语法说明）
    │   ├─ 构建 user prompt（字段信息 + 示例数据 + 查询需求）
    │   ├─ 调用 DeepSeek chat API
    │   └─ 验证 SQL 安全性（只允许 SELECT）
    ├─ explain_code(sql)                  → 解释 SQL 逻辑
    └─ optimize_query(query)              → 优化自然语言表达
```

**Prompt 结构拆解：**

```
System: 你是审计数据分析专家...
        使用 DuckDB SQL 语法...
        只生成 SELECT 查询...
        表名 "data"（序时账）和 "balance_data"（余额表）...

User:   字段列表: 日期(VARCHAR), 科目名称(VARCHAR), 金额(DOUBLE)...
        示例数据: [{"日期":"2024-01-01", ...}, ...]
        查询需求: [用户输入的自然语言]
```

### 4.3 `modules/preset_rules.py` — 预设筛选规则

规则以"规则包（pack）"组织，支持内置包和自定义规则。

```
数据结构:
  {
    "version": 2,
    "packs": [
      {"id": "general",  "name": "通用规则",   "source": "内置", "rules": [...]},
      {"id": "industry", "name": "行业特征包", "source": "行业经验", "rules": [...]},
      {"id": "ey_gam",   "name": "EY GAM",     "source": "EY GAM",  "rules": [...]}
    ],
    "custom_rules": [...]
  }

每条规则:
  {
    "id": "integer_amount_detection",
    "name": "整数金额检测",
    "description": "检测金额为整数的记录...",
    "category": "数据质量",
    "sql_template": "SELECT ...",
    "params": [{"name": "amount_field", "label": "金额字段", "type": "field_selector", "default": "金额"}]
  }
```

**API：**
- `GET  /api/preset-rules` — 扁平列表（兼容旧前端）
- `GET  /api/preset-rules/packs` — 规则包分组结构（前端主用）
- `POST /api/preset-rules` — 新增/更新自定义规则（写入 custom_rules）
- `DELETE /api/preset-rules/<id>` — 删除自定义规则

**扩展方式：** 新增行业包直接在 JSON 的 `packs` 数组里加一项即可，前端自动渲染。

### 4.4 `modules/sampling.py` — 审计抽样

4 种抽样方法，每种生成对应的 DuckDB SQL：

| 方法 | 逻辑 | 适用场景 |
|------|------|---------|
| 货币单位抽样 (MUS) | 按金额降序排，取前 N 条 | 高金额优先抽中 |
| 随机抽样 | `ORDER BY random() LIMIT N` | 无偏抽样 |
| 分层抽样 | 按字段分组，每组等比例取 | 跨科目均匀抽样 |
| 定向选取 | 金额 > 阈值，按金额降序取前 N | 大额异常排查 |

### 4.5 `modules/mapping_history.py` — 映射历史

```
存储位置: temp/mapping_history.json
索引键:   MD5(文件名 + 文件大小)

流程:
  保存: 字段映射配置成功 → 计算 key → 写入 JSON
  查找: 上传文件时 → 计算 key → 查找是否已有映射记录
  应用: 命中历史 → 前端弹 banner → 用户确认 → 预填映射

注意: 只匹配文件名+大小完全一致的，避免张冠李戴。
```

### 4.6 `modules/audit_trail.py` — 审计轨迹

```
存储位置: temp/audit_trail.jsonl
格式:     每行一个 JSON 对象，追加写入
事件:
  ├─ generate_code  → 用户查询 + 生成的 SQL + 成功/失败
  └─ execute_sql    → SQL + 影响行数 + 执行时间 + 成功/失败
```

**为什么不轮转？** 审计数据量小，一个审计项目通常几百条查询，JSONL 足够。如果未来需要轮转，可以在启动清理逻辑里加。

### 4.7 `modules/integrity_checker.py` — 完整性测试

3 条测试，全部通过 DuckDB SQL 执行：

1. **序时账完整性** — 按凭证号+日期分组汇总，检查借贷平衡
2. **科目余额表完整性** — 期末-期初=发生额
3. **交叉验证** — 序时账与科目余额表 FJ 核对

## 五、前端结构

### 5.1 页面路由

| 路由 | 页面 | 说明 |
|------|------|------|
| `/upload` | 上传页 | 文件选择、上传、历史映射匹配 |
| `/field-mapper` | 字段映射页 | 标准字段 vs 用户字段映射配置 |
| `/api-config` | API 配置页 | 输入 DeepSeek API Key |
| `/integrity-test` | 完整性测试页 | 3 条自动化测试 |
| `/query` | 查询页 | 自然语言查询 + SQL 编辑器 + 结果表格 |

### 5.2 页面内容

**上传页 (`upload.html`)**
- 拖拽/点击上传 CSV 或 XLSX
- 上传后自动检测字段、显示预览
- 若检测到映射历史，显示 banner 提示是否应用

**字段映射页 (`field-mapper.html`)**
- 智能推荐映射（基于关键词匹配 + 类型匹配 + 置信度评分）
- 置信度图标：高(绿✓) / 中(黄⚠) / 低(红✗)
- "仅显示低置信度"筛选 checkbox
- 支持序时账 + 科目余额表双表映射

**查询页 (`query.html`)**
- 侧边栏：数据概览、字段列表、查询示例、查询历史
- 筛选规则包：3 个规则包折叠展示，点击"应用"填入自然语言输入框
- 自然语言输入 → AI 生成 SQL → 在 DuckDB 执行 → 表格渲染 + 分页
- CodeMirror SQL 编辑器（默认收起，生成代码时自动展开）
- 结果导出 CSV / XLSX
- 预设规则管理弹窗（查看、新增、删除自定义规则）

## 六、Session 数据结构

Flask session 中存储的内容（Flask-Session，文件系统，序列化到 flask_session/ 目录）：

```python
session = {
    'session_id': str(uuid),           # 会话唯一 ID
    'api_key': str,                    # DeepSeek API Key
    'uploaded_file': str,              # 上传文件的原始路径
    'data_info': {                     # 序时账数据信息
        'file_path': str,
        'fields': [{'name', 'type', 'sample'}],
        'preview': [{...}],
        'mapped_fields': [...],        # 映射后的字段
        'mapped_preview': [...],
    },
    'balance_data_info': {...},        # 科目余额表信息（同上结构）
    'field_mapping': {...},            # 序时账字段映射 {"标准名": "用户列名"}
    'balance_field_mapping': {...},    # 余额表字段映射
    'duckdb_imported': bool,           # 是否已导入 DuckDB
    'duckdb_path': str,                # .db 文件路径
    'mapping_prefilled': bool,         # 是否使用了历史预填映射
    'query_history': [                 # 查询历史（最多50条）
        {'query': str, 'sql': str, 'timestamp': str, 'success': bool}
    ],
}
```

## 八、启动清理逻辑

`app.py` 启动时会跑一次 `cleanup_stale_files()`：

```
删除条件: 最后修改时间 > 24 小时
删除范围:
  ├─ temp/db/*.db 和 *.db.wal  → 孤儿 DuckDB 文件
  ├─ temp/result_*.json         → 过期的查询结果缓存
  └─ temp/export_*.*            → 过期的导出文件

保留:
  ├─ mapping_history.json       → 映射历史长期保留
  ├─ audit_trail.jsonl           → 审计轨迹长期保留
  └─ 源文件 (.xlsx/.csv)         → 用户上传的原始数据
```

## 九、关键设计决策记录

### 9.1 为什么 CSV 导入时设 all_varchar=true？

DuckDB 的 CSV 自动类型推断在混合类型列上会报错。财务 CSV 经常出现"金额列里有个 '合计' 文本行"这种问题。设置全部读成文本，后续用 `CAST` 显式转换，虽然慢一点但不出错。

### 9.2 为什么保留非映射字段（Branch B）？

审计数据经常有额外字段（如"备注"、"审核人"），用户可能没映射但它们有分析价值。Branch A 会丢弃这些字段，Branch B 保留。当前代码默认走 Branch B。

### 9.3 为什么前端不用框架？

这个工具总共 5 个页面，用 React/Vue 需要 npm install、webpack 构建、路由配置，对审计员来说修改门槛太高。原生 HTML 改哪里保存哪里，刷新即生效。

### 9.4 为什么 CodeMirror 初始 display:none 要做 refresh？

CodeMirror 在 `display:none` 的容器里初始化时无法计算自身尺寸。展开时必须调用 `codeMirrorEditor.refresh()` 让 CodeMirror 重新测量。需要在 `setTimeout(100ms)` 后调用，等浏览器先渲染完展开动画。

## 十、常见问题

### Q: 启动报错 "Address already in use"
A: 上一个进程没关干净。运行 `pkill -f "python app.py"` 再试。

### Q: 上传 CSV 后字段都变成了一列
A: 检查 CSV 的分隔符和引号。DuckDB `read_csv` 默认用逗号分隔、双引号转义。如果是 Tab 分隔的 `.tsv`，需要改代码。

### Q: DeepSeek 生成 SQL 太慢
A: 检查网络。首次请求有冷启动延迟（~2-3秒），后续会快一些。可以试试在 DeepSeek 对话里先测试 API Key 是否正常。

### Q: 自定义规则保存后刷新丢了
A: 检查 `temp/preset_rules.json` 文件权限。自定义规则写入这个文件，如果应用没有写入权限会静默失败。
